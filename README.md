India's **7 million+ food delivery workers** on Zomato and Swiggy are the backbone of the digital economy. But every time an external disruption hits — heavy rain, a curfew, extreme pollution, a platform outage — they lose **20–30% of their monthly earnings** with zero protection.

> *"Currently, gig workers have no income protection against these uncontrollable events. When disruptions occur, they bear the full financial loss with no safety net."* — DEVTrails Problem Statement

|---|---|---|
| **Environmental** | Extreme heat, Heavy rain, Floods, Severe pollution | Cannot work outdoors / Deliveries halted |
| **Social** | Unplanned curfews, local strikes, sudden market/zone closures | Inability to access pickup/drop locations |
| **Platform** | App crashes, server outages, GPS failures | Orders cannot be accepted or completed |

</div>

## Our Solution

**EliteSuraksha** is a **parametric income insurance platform**. Instead of asking workers to prove what they lost, we pre-define objective trigger conditions. When a trigger fires, the payout happens automatically — no forms, no waiting.

```
╔══════════════════════════════════════════════════════════════╗
║  TRADITIONAL INSURANCE                                       ║
║  Event → File Claim → Investigation → Assessment → Payout   ║
║  Timeline: 7–30 days                                         ║
╠══════════════════════════════════════════════════════════════╣
║  ELITESURAKSHA (PARAMETRIC)                                  ║
║  Event → AI Detects → Validates → Fraud Check → Payout      ║
║  Timeline: < 2 minutes                                       ║
╚══════════════════════════════════════════════════════════════╝
```

**Three core principles:**
1. **Automate everything** — zero human touchpoints in the claim pipeline
2. **Use objective data** — payouts based on real sensor/API data, not self-reported loss
3. **Weekly pricing** — matches how gig workers actually earn and budget

**Coverage is strictly limited to income loss.** The following are explicitly stored with every policy and enforced by the system:

| Excluded Coverage | Why |
|---|---|
| Health insurance | Out of scope — income loss only |
| Life insurance | Out of scope — income loss only |
| Accident reimbursement | Out of scope — income loss only |
| Vehicle repair | Out of scope — income loss only |
| Non-trigger-linked claims | No payout without a valid parametric event |
| Duplicate same-event claims | System blocks at DB level |
| Claims outside covered area/window | Geofencing + coverage window enforced |

### Chosen Persona: Food Delivery Partners — Zomato & Swiggy

We chose food delivery specifically because:
- Highest outdoor exposure to environmental disruptions of any gig category
- Earnings are shift-based — a 2-hour rain event = direct, measurable income loss
- Dense urban operations make disruption signals measurable in real-time
- Most digitally accessible segment (smartphone + app-native workers)

### Golden Rules Compliance

The DEVTrails problem statement defines three non-negotiable constraints:

| Golden Rule | Requirement | Our Compliance |
|---|---|---|
| **1. Persona Focus** | Must choose one delivery sub-category | ✅ Food delivery — Zomato & Swiggy |
| **2. Coverage Scope** | INCOME LOSS ONLY. No vehicle repair, health, or accident cover | ✅ All payouts are wage-replacement only. Exclusions **stored in DB** with every policy. |
| **3. Weekly Pricing** | Premium model must be structured on a weekly basis | ✅ Weekly cycle — DB-tracked, formula-driven, deducted via UPI AutoPay |

## How It Works

```
1. WORKER ONBOARDS
   Phone number entered → OTP generated + stored as hash in DB
   → OTP verified → JWT issued → Profile created in PostgreSQL
   → KYC/employment verification started → Documents uploaded (Multer)
   → Risk score computed and persisted → Weekly premium quoted → Policy activated

2. REAL-TIME RISK MONITORING (Risk Engine)
   Multi-factor risk computed from city, platform, KYC status, autopay status
   → Weather risk + AQI risk + Heat risk + Flood risk → Total risk score (0.0–1.0)
   → Recommended premium derived → Snapshot persisted to DB

3. PARAMETRIC TRIGGER FIRES
   Trigger event ingested (heavy rain / AQI / heat / flood / curfew)
   → Active policies in matching city identified from DB
   → Coverage window validated
   → Claim auto-created — one per worker per event (duplicates blocked at DB level)

4. FRAUD SCORING (parallel, < 5 seconds)
   Policy state check + risk snapshot check + claim frequency check
   + KYC/employment status + event severity signal
   → Fraud score computed → Decision: auto-approve / manual review / reject

5. PAYOUT PROCESSING
   Auto-approved claim → Payout record created in DB
   → Claim status → PAID → Worker dashboard updated
   → Admin payout pipeline updated
   → Worker receives push notification: "₹600 credited" in < 2 minutes
```

## Backend Design

The backend follows strict separation of concerns:

| Layer | Responsibility |
|---|---|
| **Routes** | Endpoint mapping — maps URL + method to controller |
| **Controllers** | Request/response handling — thin, no business logic |
| **Services** | All business logic — risk, pricing, claims, fraud, payouts |
| **Middlewares** | Auth, admin guard, validation, centralized error handling |
| **Utils** | Shared helpers — OTP generation, token creation, formatting |
| **Prisma** | All DB access — type-safe, migration-managed |

### Main Backend Modules

| Module | Purpose |
|---|---|
| `auth` | OTP generation, verification, JWT issuance, session management |
| `worker` | Profile creation, KYC/employment state, document upload |
| `policy` | Quote generation, policy activation, exclusions storage, renewal |
| `risk` | Multi-factor risk computation, snapshot persistence, premium derivation |
| `trigger` | Parametric event ingestion, policy matching, claim auto-creation |
| `claim` | Claim lifecycle management, fraud scoring trigger, status updates |
| `payout` | Payout creation, processing, status tracking |
| `admin` | Dashboard metrics, worker verification, claim review, payout processing |

**Key Design Principles:**
- Every payout requires a prior auto-approved claim — no shortcut paths
- Duplicate same-event claims blocked at database constraint level
- All exclusions stored as a text snapshot with every policy record
- Admin-only endpoints protected by role-based middleware guard
- All errors handled centrally — consistent error format across all endpoints

## Database Design

The database is designed with Prisma schema and PostgreSQL. All models are version-controlled via migrations.

### Main Tables / Models

| Model | Purpose |
|---|---|
| `users` | All platform users — worker and admin roles, status tracking |
| `worker_profiles` | Worker-specific onboarding state, KYC status, employment status, city, platform |
| `worker_documents` | Uploaded KYC, employment, and proof documents with file paths |
| `policies` | Weekly coverage, premium amount, coverage dates, status, exclusion snapshot |
| `trigger_events` | Parametric external events — type, city, severity, timestamp |
| `claims` | Auto-created claims linked to policies + triggers, fraud score, status |
| `payouts` | Payout records linked to approved claims, amount, status |
| `risk_snapshots` | Computed risk components + recommended premiums over time |
| `location_logs` | Worker GPS data for validation and fraud checks |
| `otp_requests` | Hashed OTP records with expiry and verification timestamps |

### Schema Relationships

```
users
 └── worker_profiles
      ├── worker_documents (many)
      ├── policies (many)
      │    └── claims (many, linked to trigger_events)
      │         └── payouts (one per approved claim)
      └── risk_snapshots (many)

trigger_events ──── claims (one trigger → many claims across workers)
otp_requests ───── users
```

## Core Business Flows

### Flow 1: Worker Login & Authentication

```
1. Worker enters mobile number
2. OTP generated → saved as bcrypt hash in otp_requests table
3. Worker submits OTP → verified against hash
4. JWT access token + refresh token issued
5. User status updated to ACTIVE
6. Protected endpoints now accessible via Bearer token
```

### Flow 2: Worker Onboarding

```
1. Worker saves profile → worker_profiles row created
2. Worker starts KYC/employment verification → status fields updated
3. Worker uploads documents → saved via Multer → worker_documents row created
4. Profile completion tracked — minimum state required for policy activation
```

### Flow 3: Risk Scoring & Premium Calculation

```
1. Risk engine pulls worker city, platform, KYC status, autopay status
2. Computes: weatherRisk + aqiRisk + heatRisk + floodRisk
3. Produces totalRiskScore (0.0 – 1.0) + recommendedPremium (₹)
4. Snapshot persisted to risk_snapshots with timestamp
5. Quote shown to worker — explainable, not a black box
```

### Flow 4: Policy Activation

```
Preconditions checked:
  ✓ User authenticated
  ✓ Worker profile exists
  ✓ KYC verification started
  ✓ Employment verification started
  ✓ No overlapping active policy exists

Policy created with:
  - planName, weeklyPremium, coverageStart, coverageEnd
  - status: ACTIVE
  - exclusions: stored as full text snapshot in DB
```

### Flow 5: Trigger → Auto Claim

```
1. Trigger event created (heavy rain / AQI / heat / flood / curfew)
2. Active policies in matching city fetched from DB
3. Coverage window validated (trigger timestamp within policy period)
4. Claim auto-created for each eligible worker
5. Duplicate claim for same worker + same event → blocked at DB level
```

### Flow 6: Claim → Fraud Score → Payout

```
1. Claim evaluated by fraud engine
2. Fraud signals checked:
   - Policy state validity
   - Risk snapshot presence
   - Claim frequency in last 7 days
   - KYC/employment completion status
   - Trigger event severity
3. Fraud score computed (0.0 – 1.0)
4. Decision applied:
   - < 0.35  → AUTO-APPROVED → eligible for direct payout
   - 0.35–0.64 → MANUAL REVIEW → admin action required
   - ≥ 0.65  → REJECTED
5. Auto-approved claim → Payout record created → Claim status → PAID
```

### Flow 7: Admin Review

```
Admin can:
→ View platform metrics (total workers, active policies, claims, payouts)
→ Inspect any worker profile + documents
→ Update worker KYC/employment verification status
→ Approve or reject manual-review claims
→ Trigger payout processing for auto-approved claims
→ Monitor full payout pipeline with status
```

## Weekly Premium Model

> **Why Weekly?** As required by DEVTrails — gig workers earn week-to-week. A weekly premium cycle matches their cash flow and builds habit without large upfront commitment.

```
Weekly Premium = Base (₹20) × (1 + Risk Multiplier) × Loyalty Discount × Tier Factor

Risk Multiplier = Weather(0.40) + AQI(0.25) + Heat(0.20) + Flood(0.15)
```

| Component | Range | Notes |
|---|---|---|
| Base Premium | ₹20/week | Baseline for all workers |
| Weather Risk | 0.0 – 0.50 | Peaks during monsoon (Jun–Sep) |
| Pollution/AQI Risk | 0.0 – 0.30 | Higher in North India winter |
| Heat Risk | 0.0 – 0.25 | Summer peak (Mar–Jun) |
| Flood Risk | 0.0 – 0.25 | Monsoon + NDMA alerts |
| Loyalty Discount | 0% – 15% | 3% off per continuous quarter |
| Standard Tier (1.4×) | Max ₹600/event | Default tier |
| Premium Tier (1.9×) | Max ₹1,000/event | Higher coverage |

**Worked Example:**
```
Low Risk (Pune, Summer):     ₹20 × (1 + 0.11) × 0.94 × 1.4 = ₹29/week
High Risk (Mumbai, Monsoon): ₹20 × (1 + 0.42) × 1.00 × 1.4 = ₹40/week
```

> **AI Enhancement (Phase 3):** XGBoost model will incorporate 47 real-time variables — replacing the current formula with a fully learned model that charges ₹2–3 less for workers in historically flood-safe zones (hyper-local 500m × 500m grid pricing, as suggested in DEVTrails tips).

## Risk Engine

The risk engine is the intelligence heart of the platform. It computes a continuous, explainable risk score for each worker based on their specific context.

### Risk Components

| Signal | Weight | Source |
|---|---|---|
| Weather Risk | 40% | Worker city + current conditions |
| AQI Risk | 25% | City air quality index |
| Heat Risk | 20% | Temperature vs threshold (42°C) |
| Flood Risk | 15% | NDMA alert status for city |

### Inputs Used

| Input | Role |
|---|---|
| Worker city | Determines environmental risk |
| Work platform | Zomato / Swiggy — affects exposure profile |
| AutoPay enabled | Risk reduction signal |
| KYC status | Completed = lower administrative risk |
| Employment status | Verified = lower fraud risk |

### Output

```json
{
  "weatherRiskScore":   0.35,
  "aqiRiskScore":       0.20,
  "heatRiskScore":      0.10,
  "floodRiskScore":     0.05,
  "totalRiskScore":     0.70,
  "recommendedPremium": 38
}
```

> Every snapshot is **persisted to the database** — making pricing auditable, explainable, and ready for ML model training in Phase 3.

## Parametric Triggers & Claim Engine

All triggers are based on **objective, third-party measurable data** — not self-reported loss. Coverage is for **income loss only**.

### Supported Trigger Types

| # | Trigger | Threshold | Source | Payout (Standard) |
|---|---|---|---|---|
| 1 | Heavy Rain | Rainfall > 50mm/hr | OpenWeatherMap + IMD | ₹300 – ₹600 |
| 2 | Flood Alert | NDMA Orange/Red alert | NDMA official API | ₹500 – ₹900 |
| 3 | Hazardous AQI | AQI Index > 300 | CPCB + WAQI | ₹200 – ₹500 |
| 4 | Extreme Heat | Temperature > 42°C | OpenWeatherMap | ₹200 – ₹400 |
| 5 | Curfew / Bandh | Official order detected | NewsAPI + Govt portal | ₹300 – ₹600 |
| 6 | Platform Outage | Downtime > 2 hours | HTTP synthetic probes | ₹200 – ₹400 |

### Claim Auto-Creation Logic

```
When a trigger event is ingested:

1. All ACTIVE policies with city = trigger.city are fetched from DB
2. Coverage window is validated — trigger.timestamp within [coverageStart, coverageEnd]
3. Claim is auto-created for each eligible worker
4. Duplicate prevention: same workerId + same triggerId → blocked at DB constraint level
5. Claim enters fraud scoring queue immediately
```

### Multi-Signal Confirmation (Phase 3)

| Trigger | Required Confirmation |
|---|---|
| Rain | Weather API + worker GPS in affected zone |
| Curfew | 2 of 3 sources: News + Govt portal + Twitter geo |
| Platform Outage | All 3 HTTP probes from different regions fail |
| Flood | At least 2 official sources (NDMA + IMD) confirm |

## Fraud Detection & Payout Engine

### Fraud Scoring Signals

| Signal | Detection Method |
|---|---|
| Inactive/suspicious policy | Policy status check at claim time |
| Missing risk snapshot | No snapshot → elevated score |
| High claim frequency | > N claims in last 7 days → flagged |
| Incomplete KYC/employment | Unverified workers → caution signal |
| Trigger severity | Extreme severity events → additional scrutiny |

### Decision Thresholds

```
Fraud Score < 0.35      →  AUTO-APPROVED  →  Payout can be processed immediately
Fraud Score 0.35–0.64   →  MANUAL REVIEW  →  Admin must approve/reject
Fraud Score ≥ 0.65      →  REJECTED       →  No payout
```

### Payout Rules

```
Only AUTO-APPROVED claims can be processed for payout
→ Payout record created in payouts table
→ Claim status updated to PAID
→ Payout amount = trigger.payoutAmount or policy tier max (whichever is lower)
→ Worker payout history updated in dashboard
```

> **Phase 3 Enhancement:** Isolation Forest ML model will replace rule-based fraud scoring. GPS spoofing detection, behavioral anomaly modeling, and peer-comparison validation will be added.

## AI/ML Integration

Per DEVTrails technical requirements — AI/ML is integrated across three functions. Phase 2 delivers real, DB-backed intelligent systems. Phase 3 upgrades them to full ML models.

### 1. Dynamic Risk Assessment & Premium Calculation

| Detail | Phase 2 | Phase 3 |
|---|---|---|
| **Model** | Multi-factor formula (4 signals, weighted) | XGBoost Gradient Boosted Trees |
| **Input** | City, platform, KYC status, autopay | 47 real-time variables |
| **Output** | Risk score 0.0–1.0 + recommended premium | Same + hyper-local 500m grid pricing |
| **Storage** | Every snapshot persisted to DB | Training data accumulates automatically |

### 2. Predictive Risk Forecasting (Phase 3)

- **Model:** LSTM (Long Short-Term Memory) neural network
- **Input:** 72-hour rolling time-series of environmental data
- **Output:** Probability of trigger breach in next 1hr / 3hr / 6hr
- **Use case:** Pre-emptive coverage escalation — system warns workers before triggers fire

### 3. Intelligent Fraud Detection

| Phase | Approach |
|---|---|
| Phase 2 | Rule-based scoring on 5 signals with configurable thresholds |
| Phase 3 | Isolation Forest anomaly detection on claim patterns + GPS spoofing detection + behavioral ML |

### 4. Data Flywheel

> Every risk snapshot, claim, payout, and trigger event persisted in Phase 2 becomes **training data** for Phase 3 ML models. We are building the dataset now.

## Troubleshooting

**Backend does not start:**
```
✓ Check .env exists in apps/api
✓ DATABASE_URL is correct and PostgreSQL service is running
✓ npx prisma migrate dev has been run successfully
✓ Port 4000 is not in use by another process
```

**"No workspaces found" error:**
```
✓ Root package.json must contain "workspaces": ["apps/*", "packages/*"]
✓ Each app must have its own package.json
```

**Web app shows "Network Error":**
```
✓ Backend is running on port 4000
✓ apps/web/.env has correct VITE_API_BASE_URL
✓ Restart Vite after changing .env (Vite does not hot-reload env changes)
```

**Mobile app cannot connect to backend:**
```
✗ Do NOT use localhost — mobile device cannot resolve localhost to your laptop
✓ Use your laptop's Wi-Fi IPv4 address (e.g., 192.168.1.105)
✓ Set EXPO_PUBLIC_API_BASE_URL=http://192.168.X.X:4000/api/v1
✓ Both devices must be on the same Wi-Fi network
```

**Policy activation fails:**
```
✓ Worker profile must exist
✓ KYC verification must have been started
✓ Employment verification must have been started
✓ No active policy already exists for this worker
```

**Payout processing fails:**
```
✓ Claim must first be evaluated (fraud scoring run)
✓ Claim status must be AUTO_APPROVED (fraud score < 0.35)
✓ Manual review or rejected claims cannot be directly paid
```

**Prisma migration errors:**
```
✓ Run: npx prisma migrate reset (warning: drops all data)
✓ Then: npx prisma migrate dev --name init
✓ Re-seed admin: node scripts/seed-admin.js
```
